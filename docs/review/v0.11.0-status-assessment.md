# HL-Mem v0.11.0 状态评估

> 评估日期：2026-07-24  
> 评估基线：`main` / `7619125`，版本 `0.11.0`，migration 022  
> 范围：只评估，不修改实现、配置、CHANGELOG 或 README

## 结论摘要

v0.11.0 的主体目标已经完成：Hindsight 对标中 P0/P1 项目基本落地，中文 claims FTS 已切换为 trigram，错误的共享 FTS query sanitizer 已按 tokenizer 拆分，`fts_tokenizer` 假配置也已删除。复跑全量离线测试得到 **325 passed、1 skipped**；没有发现应阻止当前单机服务继续运行的已确认功能 bug。

当前主要问题不是缺少另一轮架构扩张，而是**发布完整性和质量门禁没有跟上实现**：

1. README、HANDOFF、CHANGELOG 与当前代码事实不一致；README 仍展示 v0.10.1、21 migrations、旧测试数和 14 routes，甚至把尚不存在的 PostgreSQL 后端标为已完成。
2. migration 022 缺少从 021 存量库升级的正式回归测试。
3. 中文 FTS 评测依赖本地 `var/hl_mem.db`，缺库时跳过，而且只打印指标、不设断言；因此它是诊断脚本，不是防回退门禁。
4. CI 只运行 `tests/unit/`，没有覆盖 integration、eval 基础设施及 scenarios；本地“325 passed”与 CI 实际门禁“292 passed”口径不同。
5. `TextSearchBackend` / `VectorSearchBackend` 当前是有价值的类型边界，但还不是运行时可替换后端；相关文档应避免把“协议已定义”写成“后端已插件化”。

建议把下一个版本定位为 **v0.11.1 发布完整性与检索回归加固**，不启动 P2-8/P2-9/P2-10 的重型实现。

### 1.2 先前 trigram 审查问题的关闭状态

`docs/review/trigram-review.md` 是对提交 `11f2ec4` 的中间态审查，不能直接当作当前缺陷清单。后续提交已经关闭其中最重要的实现风险：

- `3e75624` 将 query sanitizer 按 `trigram` / `unicode61` 拆分，并统一做 FTS5 phrase quoting；`events_fts` 不再受到 claims trigram 语义污染。
- `7619125` 删除了不能控制 schema 的 `fts_tokenizer` 环境配置，恢复“migration 决定 tokenizer”的单一事实源。
- 当前单元与全量测试均通过，没有复现当时的 FTS 语法回退。

仍未关闭的是该审查提出的**升级回归测试、发布说明和容量/排序语义记录**，这些应保留为 v0.11.1 工作。

### 1.3 技术债与能力边界

#### 检索后端协议仍是演进边界

`TextSearchBackend` 和 `VectorSearchBackend` 已建立结构化协议，`ClaimRepository` 可满足协议；这是正确的低侵入准备。但 staged pipeline 仍直接使用 repository，尚无文本后端工厂或注入点。`HL_MEM_VECTOR_BACKEND` 目前只被读取、展示并测试默认值，没有驱动实现选择。

因此当前准确表述应是：**已建立后端协议边界，默认实现仍固定为 SQLite FTS5 + Python 全扫向量**。这不是当前 bug，但若文档声称“可运行时切换后端”，就会成为能力误报。`vector_backend` 与已删除的 `fts_tokenizer` 不完全相同——前者可以作为明确的未来选择器保留——但在增加第二实现之前，应校验只接受 `sqlite_scan`，或明确标注为只读 capability，避免无效环境值制造假切换。

#### 可观测性仍有下一层缺口

v0.11.0 已有 LLM span 账本、24h 聚合、job stage/progress/heartbeat 和最近一次向量候选数，足以完成本轮对标目标。尚未实现但可继续观察的项目包括：

- audit dropped count 是否已稳定暴露到服务 health/stats，而不只是 `AuditLogger.health()`；
- dense 阶段的持续 P95 与 active vector count 阈值告警，而非仅有“最近一次候选数”和手工 benchmark；
- LLM provider 的非阻塞健康状态、连续失败数和最近成功/错误时间。

这些属于运维增强，不应与 v0.11.0 发布修订混在一起。

## 2. CHANGELOG 评估

**需要更新，而且属于发布必做。** 当前条目记录的是切换前的发现和协议占位，不是最终交付状态。建议在 v0.11.0 中增加一个“中文 FTS 修复与 migration 022”小节，至少覆盖：

- `claims_fts` 与 `claims_tags_fts` 从 unicode61 重建为 trigram，并对存量数据回填；`events_fts` 继续使用 unicode61。
- 中文连续子串恢复可检索；trigram 的全文查询片段至少需要 3 个 Unicode 字符，1–2 字符片段不产生命中。
- query sanitizer 按 tokenizer 分流并进行字面量 quoting，修复标点、FTS 操作符和 `events_fts` 兼容问题。
- 删除无效的 `fts_tokenizer` Settings/环境变量；tokenizer 是 schema capability，只由 migration 决定。
- tokenizer 改变会改变 BM25 统计和候选顺序，不承诺与 unicode61 排序相同。
- 现有 514 claims 微基准中 trigram FTS 页占用约为 unicode61 的 4.4 倍、查询仍低于 1 ms；该数字应明确标注为样本观测，不是容量保证。
- migration 022 会在单一写事务中 drop/recreate/backfill 两张 FTS 表；大库升级前需评估启动耗时、写阻塞和临时磁盘/WAL 空间。
- 数字修正为 325 passed、1 skipped、22 migrations；如果 CHANGELOG 坚持只列新增测试数，也要与可复现的测试命令口径一致。

同时删除或改写“`fts_tokenizer` 配置占位”，因为该字段已经被明确认定为假配置并移除。

## 3. 快速收益改进

按收益/侵入比排序：

1. **修正文档事实源。** 更新 README、CHANGELOG、HANDOFF；顺手校正 architecture 中 PostgreSQL 现状。这是最低风险、最高确定性的收益。
2. **补 migration 022 升级回归。** 用测试构造 021 schema + 存量 claim，再启动当前 `Database`，验证数据、索引、trigger 与 `schema_migrations`。这是最重要的代码侧快速收益。
3. **把 trigram 行为做成确定性测试。** 覆盖至少：三字及以上中文、中文子串、中英混合、`C++`/`foo-bar`/路径/email/引号、1–2 字符、events unicode61。特殊字符测试必须断言期望命中，而不只是“不抛异常”。
4. **扩大 CI 的离线覆盖面。** 增加 integration；将不依赖真实数据库的 eval dataset/metrics/runner 测试纳入 CI。真实 API、真实生产快照和 benchmark 保持显式任务。
5. **为中文 FTS 建立发布基线。** 将 30 cases 拆成可提交的小 fixture 契约集与可选生产快照评测集：前者阻止功能回退，后者衡量真实召回质量。不要用同一个跳过测试同时承担两种职责。
6. **收紧 `HL_MEM_VECTOR_BACKEND` 契约。** 当前只有 `sqlite_scan` 时，非法值应在 Settings 校验阶段失败；等第二后端出现再开放 registry。
7. **记录一次生产副本迁移数据。** 保存 migration 022 耗时、`page_count`、`freelist_count`、文件大小和典型 FTS p50/p95，作为未来容量阈值的第一条真实基线。

不建议把新增 boost、召回通道或排序权重列为快速收益。v0.10.1 已明确冻结排序因子，当前首先应让 trigram 的单变量变化得到可靠评测。

## 4. Hindsight 建议的下一版本取舍

### 应在 v0.11.1 做

- 完成本评估列出的发布文档同步。
- 为 migration 022 和 tokenizer 分流补回归测试。
- 把中文 FTS 的一部分用例转为确定性质量门禁。
- 对协议边界、运行时实现和配置 capability 做准确命名与校验。
- 扩展 CI 的离线测试层，统一“测试通过数”的口径。

这些工作是 v0.11.0 已实现能力的收口，不会引入新的架构变量。

### 继续暂缓 P2-8：向量后端替换 / ANN

维持原触发条件：单 namespace 活跃向量达到约 10k，或 dense P95 持续超过 50 ms，再评估 sqlite-vec/FAISS；出现多进程共享或服务化规模后再评估 pgvector。当前已具备协议和 benchmark，缺的是持续指标证据，不是另一套存储实现。

下一版可以做的只有低侵入准备：明确 `embedded_candidate_count` 的含义、记录 dense duration，并形成可重复的阈值报告。不要现在迁移 PostgreSQL。

### 继续暂缓 P2-9：分层配置 + MemorySpace 强隔离

触发条件仍应是出现第二个真实的安全/配置隔离单元。目前 `namespace` 是软作用域，不应包装成强租户承诺。下一版最多做文档语义澄清和配置字段作用域清单，不实现数据库覆盖、继承、热加载或独立索引。

如果产品路线确认近期要多用户，必须先做完整的隔离威胁模型和跨 namespace 查询审计；不能只在现有字段上增加 `MemorySpace` 命名。

### 继续暂缓 P2-10：完整 Router / Bank / cgroup 线程管理

当前只有一个真实 reranker 后端，主要依赖远程 LLM/embedding/reranker，也没有 numpy/torch/FAISS 常驻计算。完整 Router、Bank 独立索引和 cgroup-aware 线程管理均缺少需求触发。

更合理的触发条件是：

- 至少两个 operation 需要不同模型，或 span 数据证明 primary/fallback 能显著改善成功率/成本；
- 本地模型或 FAISS 被引入，开始出现 CPU/GPU/线程争用；
- Bank/MemorySpace 已成为真实资源与授权边界。

在此之前，继续积累 LLM span 数据，并可单独设计一个很窄的 primary/fallback 策略；不要引入完整 LiteLLM Router。

## 5. README 评估

**必须更新。** README 是当前最明显的发布缺口，建议围绕“用户能否正确安装、理解和运维 v0.11.0”重写，而不是只改版本号。

### 必须修改

- 顶部状态改为 v0.11.0、325 passed、1 skipped、22 migrations。
- storage/架构图改为 migration 022，并说明 claims/tags 为 trigram、events 为 unicode61、向量仍是 SQLite BLOB 全扫。
- API 列表加入 `POST /v1/extract/dry-run` 与 `POST /v1/consolidate`，route 数从 14 改为 16。
- 增加 v0.11.0 能力：LLM call spans、job progress/heartbeat、ConsolidationScope、occurred range/entities、reranker registry、文本/向量协议边界。
- 删除“可选 PostgreSQL 后端 ✅”；改为“未来规模触发项”或直接不列。
- 删除所有旧测试数（292、284）和 21 migrations 表述。
- 配置表不要再出现 `HL_MEM_FTS_TOKENIZER`；说明 tokenizer 由 schema migration 决定。
- 在升级说明中提示 migration 022 的索引重建、短查询限制、磁盘增量和大库维护窗口。

### 建议修改

- 将“协议边界已完成”和“可替换后端已实现”明确分开。
- 补充 healthz 中 LLM 24h 聚合、job progress 和 vector candidate metrics 的说明。
- 给 dry-run extraction 与定向 consolidation 各提供一个最小 API 示例。
- 统一真实运行命令：项目规定使用 `uv`，README 测试命令应与 CI/发布口径明确对应。
- 将“当前能力”和“路线图”拆开，避免 PostgreSQL、多租户、本地模型等未来项混入完成列表。

## 建议版本范围

建议下一个版本为 **v0.11.1**，范围严格限定为：

1. 发布文档同步；
2. migration 022 与 FTS query 回归测试；
3. 中文 FTS 确定性门禁；
4. CI 离线覆盖扩充；
5. 后端 capability 配置校验和措辞收口。

完成上述收口后，再根据真实指标决定 v0.12.0 的主题。现有证据不支持把 ANN、MemorySpace 强隔离或完整多模型 Router 设为 v0.12.0 的默认目标。
