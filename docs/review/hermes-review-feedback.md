# Hermes 评审意见：对 Codex 品质审查报告的反馈

## 总体评价

报告质量很高，6.5/10 的评分和"设计完成、收敛未完成"的核心判断我完全认同。以下是我的逐条反馈，标 AGREE（完全同意）/ ADJUST（同意但需要调整）/ CHALLENGE（有不同意见）。

## 反馈清单

### 1. AGREE: 假阶段函数（品味问题 #1）

**已验证属实。** `_filter_and_score`/`_expand_related`/`_rerank`/`_finalize` 确实全部是 no-op，所有工作都在 `_collect_candidates` 里。这是 Batch 3 拆分的失败——形式上有了管线，实质上没有分布职责。

**追问**：这个问题的正确修复方向是什么？是把全部逻辑留在 `_collect_candidates` 里然后删掉假阶段（承认不需要多阶段），还是真正把工作分配到各阶段（让阶段有实际输入输出）？哪种更好取决于召回管线是否真的需要可替换的阶段边界。请给出你的建议。

### 2. AGREE: 运行时 monkeypatch 私有方法（品味问题 #2）

`api/server.py` 用 lambda 覆盖服务私有方法 `_queue_event`。确认这是坏味道。

**追问**：这是为什么这样做的？是测试需要还是运行时需要？如果是测试需要，正确做法是什么？

### 3. AGREE: 过期兼容层未清理（品质问题 #4 + 死代码报告）

v0.6.0 版本里仍有写"will be removed in v0.6.0"的兼容层。版本承诺失信。

### 4. AGREE: 同一功能重复实现（品味问题 #5）

vector 编解码两套、RRF 两套、HTTP retry 两套。这是典型的两代代码痕迹。

### 5. AGREE: 事务边界不一致

repository 默认 commit / application 传 commit=False / domain 函数自己 commit / worker 直接写 SQL。四种模式并存。

### 6. AGREE: 熔断器非线程安全

`can_call()` 没有原子门，half-open 时多线程可同时通过。对有 prefetch 线程 + async hook 的场景是真实风险。

### 7. ADJUST: 优先级排序需要调整

报告把"召回质量评测闭环"列为 #1 优先级。我认为：

- **对本地单 Agent 记忆系统**，先让代码收敛（消除两代代码、修复假阶段、统一事务契约）比建评测闭环更紧急。因为现在有两套 RRF、两套 retry、假阶段函数——在这样的代码上做评测，结果不可靠。
- 正确顺序应该是：先收敛代码（让每层只有一种正确做法）→ 再建评测闭环（在稳定代码上做基线）。
- 请重新评估优先级，考虑"哪些问题如果不先修，会导致后续所有改进都建立在不稳定的基础上"。

### 8. ADJUST: "两代代码"需要更精确的界定

报告多处提到"两代代码"但不够具体。请给出一个明确的清单：
- 哪些是"必须现在收敛"的两代代码（影响正确性或可维护性）
- 哪些是"正常的演进痕迹"（旧版本兼容层有 DeprecationWarning，可以按计划删除）

区分标准：前者是"两种实现都在活跃使用且可能产生不一致行为"，后者是"旧实现只是 re-export，新代码已经走新路径"。

### 9. CHALLENGE: docstring 和风格问题不应列入品质审查的主要发现

品味问题 #9（docstring 只复述函数名）和 #10（中英文混用、风格不统一）虽然属实，但对"精致项目"的影响远小于假阶段、monkeypatch 和事务不一致。建议把它们降级为"次要 polish 项"，不要和结构性问题混在一起。

### 10. 缺失功能分析需要补充实际优先级

报告列了 7 个缺失功能，但没有明确哪些是"现在就该做"，哪些是"下一阶段再说"。请补充：
- 这 7 个缺失功能中，哪 3 个对"让 hl_mem 成为一个精致的记忆系统"最关键？
- 其余 4 个应该排在什么时间点？
