# hl_mem v0.6.0 代码精简度审查

## 背景

hl_mem 是一个**本地单 Agent 记忆系统**（Python + SQLite），刚完成 Phase 15 大重构（14个问题修复，v0.4.3→v0.6.0）。

当前规模：95 个 .py 源文件，9543 行代码，249 测试。

## 审查目标

**核心问题：项目里有没有"大而无用"的代码？有没有为了架构完整而提前实现的僵尸功能？有没有可以删掉但没人敢删的死代码？**

这是一个本地个人记忆系统，不是企业级 SaaS 平台。每一行代码都应该有明确的实际用途。

## 审查维度

### 1. 死代码与僵尸功能
- 有没有定义了但从未被任何代码路径调用的函数/类/方法？
- 有没有定义了但从未使用的配置项/环境变量/Settings 字段？
- 有没有创建了但从未填充数据的数据库表？
- 有没有注册了但从未被实际触发的 API 端点？
- 有没有写了但从未执行的 migration 或 job type？

### 2. 提前实现（YAGNI 违反）
- 有没有为了"未来可能需要"而提前实现的接口/协议/抽象层？
- 有没有只有一种实现但定义了 Protocol/ABC 的提前抽象？（除非有明确计划）
- 有没有定义了字段但当前场景从不使用的数据模型字段？
- PostgresDatabase 是实验性探针还是有实际用途？如果不用，应该删还是保留？

### 3. 过度防御性编程
- 有没有永远不会触发的 try/except（捕获了不可能发生的异常）？
- 有没有对永远不会为 None 的值做 None 检查？
- 有没有对永远不会越界的值做边界检查？
- 有没有永远不会进入的 if/else 分支？

### 4. 冗余代码
- 有没有功能重复的函数（微调版本/旧版本共存）？
- Batch 2 的 DeprecationWarning re-export——这些兼容层是否还有调用方？如果没有外部调用，可以直接删了
- 有没有测试通过但代码实际上没人用的 utility 函数？

### 5. 功能价值评估
逐个评估以下功能模块：它是真正在被使用，还是"为了架构完整"而存在的摆设？
- Experience/Episode/Trace/Policy 系统——当前实际有 episode 数据产生吗？
- Mental Models 模块——当前有任何调用方吗？
- Relation Expansion（关系扩展召回）——默认关闭，值不值得保留？
- MCP Server——有实际调用方吗？
- Observability/Audit——audit 数据有人在看吗？还是只写不读？
- Feedback 系统——有实际反馈数据产生吗？
- Security/Retention——有实际保留策略在执行吗？

### 6. 文件大小合理性
审查每个 200+ 行的文件：
- 它的行数是否与它的职责复杂度匹配？
- 有没有可以合并到其他文件的小模块（不值得独立成文件）？
- 有没有应该拆分的大文件？

## 输出要求

1. **死代码清单**：每个标注 位置 + 原因 + 建议（删除/保留理由）
2. **僵尸功能清单**：每个标注 当前是否有实际数据/调用 + 建议
3. **精简建议**：如果能删掉 X 行代码而不损失任何实际功能，列出删除清单
4. **代码精简评分**：1-10（1=全是死代码，10=每行都有价值）
5. **给出一个数字：当前 9543 行里，估计有多少行可以安全删除？**

## 审查要求

- **读完全部 src/hl_mem/ 下源码**
- **同时检查实际数据库**：用 `sqlite3 var/hl_mem.db` 查各表行数，判断功能是否实际在用
- **检查 API 端点的实际调用**：搜索 Hermes 插件代码（adapters/hermes/）看哪些端点实际被调用
- **具体到文件:行号**
- **诚实评估**：不要因为是"架构合理"就保留。如果当前没有数据/没有调用，就是死代码

## 重要约束

**不要修改任何代码文件，只输出审查报告。**
**不要运行 pytest。**
**将报告写入 docs/review/codex-dead-code-review-report.md，然后 git add docs/ && git commit。**
