# HL-Mem v0.9.0 全面代码审查

审查日期：2026-07-24
审查范围：`src/hl_mem/` 全目录，重点检查 Phase 17 的 slot/tag、冲突、跨 subject 去重、TTL、回填、召回与配置链路。
审查性质：只读分析；未修改任何源码。

## 结论

**整体评分：4.8 / 10**

**发布结论：不可以发布。**

v0.9.0 的新增字段采用了加法迁移，默认跨 subject 去重也是 audit-only，这两个方向正确；现有测试也全部通过。但自动去重开启后存在低置信度判定直接触发数据合并的 P0 风险，TTL worker 会因 ISO 时间字符串的时区偏移而漏过期，slot 必需 qualifier 没有在任何可信边界执行，手工回填还可能覆盖并发产生的新分类数据。上述问题会改变或污染长期记忆，不能作为一般缺陷留到发布后处理。

验证结果：

- `.venv/Scripts/python.exe -m pytest tests/unit -q --tb=short`：225 passed。
- `.venv/Scripts/python.exe -m pytest tests -q --tb=short`：250 passed，1 skipped。
- 测试通过不能覆盖本报告的关键风险：`deduplicate_claims`、`backfill_expires_at`、`backfill_claim_slots_v1` 和 `compute_expiration` 没有对应的直接测试。

## P0 阻断

### P0-1：自动去重无置信度门槛，任意 `equivalent` 判定都会执行不可逆语义合并

- 文件：`src/hl_mem/workers/deduplicate.py:116-147`
- 相关文件：`src/hl_mem/domain/claims/dedup.py:64-69`、`src/hl_mem/workers/deduplicate.py:163-205`
- 描述：
  - `DedupJudge` 返回并持久化 `judge_confidence`，但应用阶段只检查 `decision='equivalent'`，完全不使用置信度。
  - 因而 LLM 返回 `equivalent, confidence=0.0` 也会在 `audit_only=False` 时 supersede 右侧 claim。
  - 应用前只重查两侧 `status='active'`，没有重验 predicate、slot、qualifier、数值/版本/路径/日期/否定护栏、来源权威性、敏感级别或受保护记忆类型。
  - judge 与 apply 之间若发生重分类，原本 `canonical_slot IS NULL` 的 claim 可以已经成为 operational slot，但旧判定仍会应用。
  - 默认 audit-only 只能降低默认部署风险，不能使公开的自动合并路径达到可发布标准。
- 修复方案：
  1. 增加独立的 `dedup_auto_merge_min_confidence`，建议首版不低于 `0.98`，并在 Settings 中校验 `threshold <= min_confidence <= 1`。
  2. 在同一 `BEGIN IMMEDIATE` 事务内重新读取 claim 和 pair，并重验：pair 仍未应用、两侧 active、predicate 相同、两侧仍无 slot、受控 qualifier 一致、值的数字/版本/路径/日期/否定护栏通过、受保护类型禁止自动合并。
  3. 将判定所依据的 claim 版本写入 pair，例如 `recorded_from`、内容哈希、slot、policy version；apply 时做 CAS。
  4. 自动动作必须显式启用，且在有标注集证明 precision 前继续保持 audit-only。

## P1 重要

### P1-1：必需 qualifier 只写在 registry 和 prompt 中，未在可信边界执行

- 文件：`src/hl_mem/domain/claims/attributes.py:509-515`
- 相关文件：`src/hl_mem/domain/claims/conflicts.py:82-91`、`src/hl_mem/ingest/schemas.py:20-31`、`src/hl_mem/ingest/extractors.py:53-66`
- 描述：
  - `validate_canonical_slot()` 只验证 slot 名是否合法，不接收 qualifiers，也不检查 `SlotDefinition.required_qualifiers`。
  - Pydantic schema 只能约束 slot enum，不能表达“选择 `config.port` 时必须有 `service`”之类的条件。
  - `slot_qualifier_key()` 对缺失字段写入 `None`。结果是所有缺少 `service` 的 `config.port` claim 会进入同一个实例键，产生伪冲突/伪去重。
  - FakeExtractor 会为 `state.service_health` 生成 operational slot，却只写 `{"state_change": True}`，没有必需的 `service`；新字段虽然初始化了，但产物不满足 registry 契约。
- 修复方案：
  - 新增统一的 `validate_slot_instance(slot, qualifiers) -> str | None`，缺少或为空的必需 qualifier 时把 slot 降为 `None`，并写 audit reason。
  - ingest、LLM 响应解析、FakeExtractor、回填和 reclassify 必须统一调用该函数。
  - 为每个带 `required_qualifiers` 的 slot 增加缺失、空串、大小写/Unicode 规范化测试。

### P1-2：冲突键包含 predicate，导致 resolver 支持的 cross-predicate 冲突在 ingest 中永远相遇不到

- 文件：`src/hl_mem/domain/claims/conflicts.py:42-51`
- 相关文件：`src/hl_mem/application/ingest.py:450-461`、`src/hl_mem/storage/claims.py:207-216`
- 描述：
  - `ConflictResolver` 按 `canonical_slot` 判断兼容性，现有单测也明确覆盖“不同 predicate、同一 slot”。
  - 但 `compute_conflict_key()` 把规范化 predicate 放进哈希；同 subject、同 slot、同 qualifier 的 cross-predicate claim 会得到不同 key。
  - ingest 只按 key 查询候选，因此 resolver 的 cross-predicate 能力在真实写入链路中不可达。
- 修复方案：
  - 若 slot 是冲突的真正单一事实来源，从 v2 key 中移除 predicate，只保留 namespace、subject、slot instance qualifier。
  - 若产品决定 predicate 必须隔离，则应删除 resolver 的 cross-predicate 语义并修正测试和设计文档；不能保留两套互相矛盾的规则。
  - 变更 key 算法必须提升版本并提供只改 key、不改 claim 内容的可恢复回填。

### P1-3：TTL worker 用文本比较时间，带非 UTC offset 的 claim 会漏过期

- 文件：`src/hl_mem/workers/ttl.py:11-24`
- 相关文件：`src/hl_mem/domain/claims/retention.py:23-28`、`src/hl_mem/application/ingest.py:400-410`
- 描述：
  - SQLite 条件是 `expires_at <= reference` 的字符串比较，不是时间比较。
  - `compute_expiration()` 保留 observed_at 的原始 offset。例如 `2026-01-02T01:00:00+08:00` 实际早于 `2026-01-01T18:00:00+00:00`，但字典序相反，claim 不会过期。
  - 这会让本应失效的状态继续参与 current recall，属于数据生命周期正确性错误。
- 修复方案：
  - 所有写入边界统一把 `observed_at`、`recorded_from`、`valid_to`、`expires_at` 规范化为 UTC 固定格式。
  - 对存量数据先做 UTC 规范化回填。
  - TTL 查询不要依赖混合格式文本的天然顺序；至少使用规范化 UTC 列，并为 `Z`、`+00:00`、`+08:00`、负 offset 增加测试。

### P1-4：slot 回填会覆盖已经存在的新字段，并存在计算与写入之间的竞态

- 文件：`src/hl_mem/storage/migrations/backfill_claim_slots_v1.py:73-129`
- 描述：
  - 脚本读取全部 claim，在事务外根据旧 `canonical_attribute` 计算更新，之后才 `BEGIN IMMEDIATE`。
  - apply 对所有行无条件覆盖 `canonical_slot`、`topic_tags_json` 和 `conflict_key`，没有 `IS NULL` 条件、版本条件或原值 CAS。
  - 若在线 ingest/reclassify 在读取后写入更准确的 slot/tags，回填会用旧 attribute 推断结果覆盖它；重复运行也会覆盖人工或新版本分类，不具备单调幂等性。
  - 整批 SQL 失败会 rollback，这一点是安全的；问题在于成功提交也可能合法地写入陈旧结果。
- 修复方案：
  - 在 `BEGIN IMMEDIATE` 内按小批读取并更新，或把读取时的原值加入 `WHERE` CAS。
  - 默认只更新 `canonical_slot IS NULL AND topic_tags_json IS NULL` 的历史行；重新分类必须使用独立命令和显式版本。
  - 保存 backfill version，并输出 `attempted/applied/cas_skipped`，不能把预计算行数当成功数。

### P1-5：expires_at 回填可覆盖并发重分类后的 permanent claim

- 文件：`src/hl_mem/workers/backfill_expires_at.py:43-90`
- 描述：
  - 查询时限定 `status='active' AND scope='temporal'`，但更新 CAS 只检查 `status='active'`。
  - 若 claim 在读后被 reclassify 为 permanent，回填仍会写入旧 temporal `expires_at`，甚至把它设为 expired。
  - 每批 commit 使脚本可恢复，但统计 `updated/expired` 在执行 SQL 前累加，无法反映 CAS 或并发跳过。
- 修复方案：
  - UPDATE 至少加 `AND scope='temporal' AND recorded_from=?`，最好同时比较读取时的 `importance/canonical_slot/expires_at` 或维护 row version。
  - 使用 `cursor.rowcount` 更新 applied/expired 统计，单独报告 CAS skipped。
  - 与 reclassify 共用一个原子仓储方法，避免两处生命周期更新规则漂移。

### P1-6：偏好召回只识别 operational preference slot，开放偏好全部失去 boost/保留配额

- 文件：`src/hl_mem/recall/staged_pipeline.py:42-44`
- 相关文件：`src/hl_mem/recall/staged_pipeline.py:63-69`、`src/hl_mem/recall/staged_pipeline.py:223-226`、`src/hl_mem/recall/staged_pipeline.py:371-375`
- 描述：
  - `_is_preference_claim()` 只查 `canonical_slot` 对应 registry definition。
  - `preference.workflow`、`preference.architecture`、`preference.other` 等被明确设计为非 operational，因此它们的 `canonical_slot` 是 `None`，即使 `canonical_attribute`/predicate 明确是偏好也会返回 False。
  - 这些 claim 在 PREFERENCE intent 下不再获得 recency boost，也不能进入偏好保留配额，构成从旧字段切换后的召回回归。
- 修复方案：
  - 偏好属于高层语义，应首先按规范化 predicate 判定；迁移期可用 `canonical_attribute` 的 registry definition 兼容兜底。
  - slot 只负责实例冲突/生命周期，不应成为 preference intent 的唯一分类来源。

### P1-7：跨 subject 候选发现无输入上限，`limit` 只截断结果，实际为全库 O(N²)

- 文件：`src/hl_mem/storage/claims.py:167-205`
- 描述：
  - 查询读取 namespace 下全部 active、NULL-slot claim。
  - 随后一次性 embedding 全部文本，并双重循环比较所有 pair；`limit` 只在排序完成后切片。
  - `dedup_scan_limit=200` 因此不是 scan limit。claim 增长后，日常 worker 会产生 O(N²) CPU/内存和 embedding API 成本，可能阻塞单 worker 的其他维护任务。
- 修复方案：
  - 用 watermark + 有界 SQL batch 限制本轮新 claim，并按 predicate 分组。
  - 使用已存 embedding 或批量向量候选索引，限制每个 claim 的 top-k 邻居；`limit` 应约束输入扫描和远程 embedding 数量。
  - 指标中分别报告 scanned claims、compared pairs、returned pairs。

### P1-8：Recall/API/MCP 输出丢弃 canonical_slot 和 topic_tags

- 文件：`src/hl_mem/application/recall.py:216-259`
- 相关文件：`src/hl_mem/api/server.py:146-174`、`src/hl_mem/mcp/server.py`
- 描述：
  - 仓储已正确解码 `topic_tags_json`，内部 claim 也携带 `canonical_slot`、`canonical_attribute` 和 `topic_tags`。
  - `_assemble_results()` 手工构造公开结果时没有输出这三个字段，REST 与 MCP 都经该服务返回，因此新元数据没有对外暴露。
  - API 没有 response DTO，契约遗漏也不会被 schema 测试发现。
- 修复方案：
  - 建立明确的 Claim 输出 DTO，兼容增加 `canonical_slot: str | None`、`topic_tags: list[str]`，迁移期保留 `canonical_attribute` 并标记 deprecated。
  - REST、MCP 和 packed context 使用同一序列化函数，并增加契约测试。

### P1-9：topic_tags 只被存储，未进入任何检索路径

- 文件：`src/hl_mem/storage/claims.py:327-337`
- 相关文件：`src/hl_mem/storage/migrations/002_claims_fts_subject.sql`、`src/hl_mem/recall/staged_pipeline.py:24-25`
- 描述：
  - `topic_tags` 被称为检索标签，但 FTS、dense 文本、query routing、ranking 均未读取它。
  - 目前它只是存储/统计字段，不能改善 recall；若 v0.9.0 对外宣称 tag 检索，这是功能缺失。
- 修复方案：
  - 明确首版契约：若仅存储，文档应去掉“用于检索”的承诺。
  - 若要支持检索，把规范化 tags 纳入独立 FTS/tag channel，再通过 RRF 融合；不要直接污染事实正文 embedding，且需离线评估 precision/recall。

### P1-10：新增 dedup cron 未在 Settings 启动校验

- 文件：`src/hl_mem/settings.py:188-214`
- 相关文件：`src/hl_mem/workers/deduplicate.py:29-40`、`src/hl_mem/workers/worker.py:168-178`
- 描述：
  - threshold、limit、TTL 和 importance 都在 Settings 校验，但 `dedup_cron` 要到 worker 调度时才解析。
  - 非法配置会让 worker 在维护调度阶段抛异常，而不是在进程启动时以明确配置错误失败。
- 修复方案：
  - Settings 统一校验严格 `HH:MM`，禁止额外字符，小时 0–23、分钟 0–59。
  - 调度函数接收已经解析的配置值，避免每轮重复解析。

## P2 建议

### P2-1：retention 纯函数对非法 scope 静默返回永不过期，且未统一 naive/aware datetime

- 文件：`src/hl_mem/domain/claims/retention.py:23-28`、`src/hl_mem/domain/claims/retention.py:43-76`
- 描述：
  - scope 不是 `permanent/temporal` 时返回 `(None, "none")`，会把调用方错误静默放大为无限保留。
  - observed_at 和 valid_to 一个无时区、一个有时区时，第 74 行比较会抛 `TypeError`，错误信息也不是领域级校验错误。
  - importance 边界本身清晰：`0.4` 和 `0.7` 都属于 normal；两个时间锚点都缺失时会明确报 ValueError。这两点无缺陷。
- 修复方案：
  - 纯函数严格拒绝未知 scope、非有限 importance 和缺失时间锚点。
  - `_parse_iso` 统一转换到 UTC，并明确项目对 naive 时间的处理规则。

### P2-2：dedup pair 的唯一键忽略 policy/text/model 版本，升级后无法自动复审

- 文件：`src/hl_mem/storage/migrations/017_dedup_pairs.sql:2-16`
- 相关文件：`src/hl_mem/workers/deduplicate.py:79-98`
- 描述：
  - `pair_key` 只由两个 claim ID 构成且全局 UNIQUE。
  - 即使 `embedding_text_version`、`policy_version` 或 judge model 改变，`INSERT OR IGNORE` 仍保留旧 decision，版本字段只审计、不参与失效。
- 修复方案：
  - 将决策版本纳入唯一约束，或提供显式的 stale/review generation 字段和重审命令。
  - `uncertain` 应有可配置的复审策略，缺失 claim 的 pending pair 应转为 terminal skipped，而不是每日重复处理。

### P2-3：JSON schema 没有限制 topic_tags 数量和唯一性

- 文件：`src/hl_mem/ingest/schemas.py:23-24`
- 描述：
  - schema 使用受控 enum，但没有 `max_length`/`maxItems` 和唯一性约束。
  - ingest 的 `normalize_topic_tags()` 会去重和过滤，因此不会直接污染库；但远端结构化输出仍可返回无界重复数组，契约与 prompt 的 0–5 个要求不一致。
- 修复方案：
  - 将字段限制为最多 5 项，并在模型校验器中去重或拒绝重复。

### P2-4：registry 尚未完全成为推断规则的单一事实来源

- 文件：`src/hl_mem/domain/claims/attributes.py:223-224`、`src/hl_mem/domain/claims/attributes.py:323-470`
- 描述：
  - allowlist、predicate map、互斥集合已从 `_SLOT_DEFINITIONS` 生成，这是正确的。
  - 但 `ATTRIBUTE_HINTS` 和 `_HIGH_CONFIDENCE_ATTRIBUTE_PATTERNS` 仍是独立手工表，registry 的 aliases/examples 没有驱动推断；新增/改名 slot 可能只更新一处。
  - 两个旧表仍被 `infer_canonical_attribute()` 实际调用，不是 dead code，也没有与新 registry 直接冲突；风险是未来漂移。
- 修复方案：
  - 对规则表增加启动期/单测一致性校验：所有目标必须存在、predicate 必须匹配 registry。
  - 能由 aliases/examples 生成的简单 hints 从 registry 生成；高置信正则可保留独立表，但必须显式引用 slot 名并验证。

### P2-5：Phase 17 关键路径缺少测试，现有全绿形成错误安全感

- 文件：`tests/`
- 描述：
  - 未发现对 `deduplicate_claims`、`backfill_expires_at`、`backfill_claim_slots_v1`、`compute_expiration` 的直接测试。
  - 缺失的关键用例包括：低置信 equivalent 不应用、judge/apply 间重分类、回填 CAS、offset TTL、缺 qualifier 降 NULL、开放 preference recall、API 新字段。
- 修复方案：
  - 在发布前补齐上述单元与 SQLite 集成测试，并加入两个连接模拟并发的测试。

## 已核验但未发现硬伤的项目

- `ExtractedClaim.topic_tags` 使用 `default_factory=list`，没有可变默认值共享问题。
- `FakeExtractor` 已初始化 `canonical_attribute/canonical_slot/topic_tags`；实际缺陷是 slot qualifier 不完整，已列为 P1-1。
- migration 016/017 都是加列/加表，没有删除 claim 数据。
- `backfill_claim_slots_v1` 在单次 `executemany` 异常时会 rollback 整批；其风险是无条件覆盖与并发陈旧写，不是异常时半提交。
- evidence 迁移使用 `INSERT OR IGNORE`，且 migration 007 已建立关系唯一索引；重复执行不会生成重复关系。
- dedup apply 在 evidence 迁移、supersede 和 `applied_at` CAS 任一步失败时会 rollback，事务边界本身正确。
- reclassify 明确跳过 expired claim，expires_at 回填和 TTL worker 也只更新 active claim；未发现直接把 expired claim 恢复为 active 的路径。
- `canonical_attribute` 仍被兼容校验、历史推断、保护类型和回填读取，当前不是 dead code；但应制定废弃版本与迁移完成条件。

## 发布前最低整改清单

1. 禁止低置信或陈旧 dedup decision 自动合并，补齐事务内护栏与 CAS。
2. 在可信边界执行 required qualifier，缺失时 slot 降为 NULL。
3. 统一 UTC 时间存储并修复 TTL 比较。
4. 修复两个回填工具的并发 CAS 和无条件覆盖。
5. 修复开放 preference 的召回识别。
6. 解决 conflict key 与 resolver 的 cross-predicate 规则矛盾。
7. 给新增 worker、回填和公开输出补齐直接测试。

完成以上项目并重新跑全量测试与迁移副本演练后，才建议进入 v0.9.0 发布候选评审。
