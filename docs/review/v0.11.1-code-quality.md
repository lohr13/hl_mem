# HL-Mem v0.11.1 源码质量审查

审查日期：2026-07-25

## 范围与方法

- 范围：`src/hl_mem/` 下 97 个 `.py` 文件，共 11,635 行；未审查 `tests/`、`scripts/`、`docs/`。
- 方法：逐文件阅读，并辅以 AST 解析、定义/引用检索、异常处理检索、类型与 docstring 扫描、行长统计、模块依赖 SCC 检查及最小行为复现。
- 限制：项目没有配置 Ruff、Mypy 等静态检查器，当前环境也未安装它们；“未使用”结论均经调用点检索复核，显式 `__all__` re-export 不计为未使用。
- 本次仅审查，不修改源码。除本报告外未改动项目文件。

## 总体结论

代码的领域边界总体清楚，`domain/core` 与 `application/storage/workers` 的方向基本合理，未发现静态循环 import。事务边界大多显式，SQL 参数也普遍使用占位符。

精致度方面仍有明显欠账：存在 4 项 P0、9 项 P1、7 项 P2。最重要的问题不是格式，而是几个“看似可降级、实际隐藏故障”的路径，以及大量 `dict[str, Any]` 在层间传播造成的契约模糊。多个兼容/占位模块和超长编排文件也增加了维护噪声。

## P0（必须修）

### P0-1：空策略 trigger 或空 query 会匹配全部策略

- 位置：`src/hl_mem/recall/recall_pipeline.py:20-35`
- 现象：`trigger in normalized_query` 在 `trigger == ""` 时恒为真；`normalized_query in trigger` 在 query 为空时也恒为真。
- 已复现：
  - `matching_policies([{"trigger": ""}], "hello")` 返回该策略；
  - `matching_policies([{"trigger": "deploy"}], "")` 也返回该策略。
- 风险：无效策略或空查询会把不相关 procedure 注入召回上下文，属于实际行为错误。
- 建议：在匹配前明确跳过空 `trigger`，并对空白 query 直接返回空列表；API 的 `RecallInput.query` 同时应增加 `min_length=1`。

### P0-2：审计组件捕获并吞掉 `BaseException`

- 位置：`src/hl_mem/observability/audit.py:123-131, 139-143, 165-172, 187-189`
- 现象：`emit()`、`cleanup()`、`close()` 捕获 `BaseException`；回滚失败也用 `except BaseException: pass`。
- 风险：`KeyboardInterrupt`、`SystemExit`、`GeneratorExit` 等控制流异常可能被当作普通审计失败吞掉，使进程无法按预期中断或退出。`span()` 虽会重新抛出，但其错误记录路径仍可能改变控制流。
- 建议：只捕获 `Exception`；回滚失败至少记录第二异常，或将其附加到 `last_error`。

### P0-3：Reranker 外部 API 没有 retry，且把失败静默降级为空结果

- 位置：`src/hl_mem/recall/reranker.py:40-72`
- 现象：embedding 与 LLM 统一使用 `retry_http()`，reranker 却单次请求；超时、429、5xx、响应结构错误全部转换为 `[]`。
- 风险：瞬时网络故障会被伪装成“没有重排结果”，调用方无法区分真实空结果与服务故障；与项目“外部 API 必须 retry + 超时”的约定冲突。
- 建议：复用 `retry_http()`，通过结构化 `RerankResult` 或受控异常向上保留失败语义；降级策略应在管线层决定，而不是 transport 层吞掉。

### P0-4：确定的死代码和未使用 import

- `src/hl_mem/api/server.py:5`：`os` 未使用。
- `src/hl_mem/components.py:22`：`FakeReranker` 未使用。
- `src/hl_mem/storage/claims.py:5`：`json` 未使用。
- `src/hl_mem/domain/claims/dedup.py:152-158`：`Deduplicator._canonical()` 在整个 `src/hl_mem` 中无调用。
- `src/hl_mem/workers/worker.py:153-155`：`Worker._dispatch()` 无调用，实际入口是模块级 `dispatch_job()`。
- `src/hl_mem/adapters/hermes/http_client.py:123-125`：`error_name()` 无调用。
- `src/hl_mem/components.py:129-146`：三组 `make_*_for_test` 与 `create_*_for_test` 在源码和测试中均无引用，且只是原样返回参数。
- 风险：制造虚假的扩展点和维护入口；其中测试工厂与别名尤其容易让调用者误以为存在额外构造语义。
- 建议：删除确定无用项；若少数方法是第三方插件反射调用的稳定接口，应以 Protocol、注册表或文档明确标注，而不是依赖“可能被外部调用”保留。

## P1（应该修）

### P1-1：核心层间契约被 `Any` 和裸 `dict` 大面积抹平

- 代表位置：
  - `src/hl_mem/application/recall.py:52-53`：`connection: Any`，且 `EmbedderProtocol | Any` 等价于 `Any`。
  - `src/hl_mem/application/ingest.py:200-202`：连接、提取结果均为 `Any`。
  - `src/hl_mem/mcp/server.py:26-27, 57, 70, 86, 97`：组件和连接均为 `Any`。
  - `src/hl_mem/recall/staged_pipeline.py:52, 57, 60-62`：管线上下文的 repo、reranker、connection、config、tracer 均为 `Any`。
  - `src/hl_mem/protocols.py:43, 46, 57, 60` 与 `src/hl_mem/storage/claims.py:288-291`：intent 为 `Any`，返回值为未参数化的 `list[dict]`。
- 影响：类型标注看似完整，但无法发现字段拼写、返回形状和组件接线错误。
- 建议：为 Event/Claim/Policy/RecallResult 建立 `TypedDict` 或 dataclass；连接统一为 `sqlite3.Connection`；intent 使用 `RecallIntent`；删除所有 `X | Any`。

### P1-2：公开 API/docstring 覆盖明显不足

- 缺少模块 docstring 的代表文件：`api/server.py`、`domain/claims/conflicts.py`、`domain/claims/dedup.py`、`ingest/budget.py`、`ingest/embedder.py`、`ingest/event_filter.py`、`ingest/extractors.py`、`ingest/llm_extractor.py`、`observability/audit.py`、`recall/ranking.py`、`recall/reranker.py`、`storage/database.py`、`workers/decay.py`、`workers/ttl.py`、`workers/worker.py`。
- 缺少有意义公开 docstring 的代表对象：
  - `api/server.py:66` `create_app`
  - `storage/claims.py:37` `ClaimRepository` 及多个公开方法
  - `ingest/budget.py:33-67` 三个公开方法
  - `observability/audit.py:74-218` 多个公开方法与 `NullAuditLogger`
  - `recall/ranking.py:28-59` 三个公开评分函数
- 影响：与项目“模块/类/公开函数必须有中文 docstring”的约定不一致，且关键行为（事务、降级、评分范围）无法从接口处确认。

### P1-3：日调度逻辑存在四份近似 copy-paste

- 位置：
  - `workers/consolidate.py:112-140`
  - `workers/induce_policies.py:67-90`
  - `workers/worker.py:44-68`
  - `workers/deduplicate.py:29-49`
- 现象：时间解析、范围校验、当天幂等键、job 构造重复；deduplicate 又改成接收 `scheduled_minutes`，形成两套调用约定。
- 影响：错误消息、时区语义和调度边界容易漂移。
- 建议：抽取一个小型 `enqueue_daily_job(connection, now, schedule, job_type, payload, env_name)`；保留业务函数作为薄命名入口。

### P1-4：HTTP 客户端策略不一致

- `http_utils.py` 有统一指数退避。
- `ingest/embedder.py`、`llm/client.py` 使用它。
- `recall/reranker.py`、`adapters/hermes/http_client.py` 未使用。
- `adapters/hermes/prefetch.py:47-49` 和 `provider.py:84-85, 177-179` 再用宽泛 `Exception` 静默降级。
- 影响：相同故障在不同组件中呈现为异常、空结果或布尔值，调用方难以形成一致恢复策略。
- 建议：统一 transport 层的 timeout/retry/error taxonomy；业务层再明确选择 fail-fast、circuit-break 或 graceful degradation。

### P1-5：超长文件和复杂编排函数仍承担过多职责

- 超过 500 行：
  - `recall/staged_pipeline.py` 612 行
  - `domain/claims/attributes.py` 599 行
  - `storage/claims.py` 552 行
  - `workers/worker.py` 550 行
  - `application/ingest.py` 513 行
- 高复杂度代表：
  - `settings.py:189-272` `_validate()`，约 36 个分支。
  - `recall/relation_expansion.py:86-197`，112 行、约 23 个分支。
  - `application/ingest.py:199-384`，186 行、约 20 个分支。
  - `recall/staged_pipeline.py:232-612` 的多个阶段函数分别达到 14–18 个分支。
- 建议：按稳定职责拆，而非机械按行数拆。优先拆出 settings 分组校验、ingest resolution/persistence、relation traversal；`staged_pipeline` 已有阶段函数，可进一步移到独立模块。

### P1-6：兼容层、占位层和真实实现混在正式源码中

- `storage/repository.py` 只有一行迁移注释，没有模块 docstring、导出或弃用机制。
- `storage/base.py` 只有说明性 docstring，没有可消费接口。
- `recall/extended_pipeline.py` 只做 re-export。
- `experience/service.py` 仅通过继承给 `ExperienceRepository` 改名，没有应用服务行为。
- `storage/postgres.py` 是实验性连接探针，却位于正式 storage 包并使用 `Any`。
- 影响：模块名承诺的职责与内容不符，读者难以判断哪些是稳定入口、兼容入口或未完成实验。
- 建议：删除无人引用的空模块；兼容模块增加明确 deprecation 注释/期限；若 `ExperienceService` 暂无业务职责，直接 re-export 或把事务编排真正移入 service；实验后端放入明确的 experimental 命名空间。

### P1-7：配置验证函数过于集中，且字符串模式没有类型化

- 位置：`settings.py:24-98, 189-272`
- 现象：`environment`、provider、mode、backend、cron 等均为普通 `str`；`_validate()` 用长串集合判断补救。
- 影响：构造后的 Settings 在静态类型上仍允许非法值；新增字段会继续放大单函数复杂度。
- 建议：对有限集合使用 `StrEnum`/`Literal`，按 LLM、recall、worker、retention 分组私有校验方法。

### P1-8：SQL 与状态字符串散落，领域枚举使用不一致

- 已有 `EpisodeStatus`、`ClaimStatus`、`RecallIntent`，但 repository/worker/API 中仍大量直接比较或写入 `"active"`、`"retired"`、`"pending"`、`"stale"`、`"claim"`、`"observation"` 等。
- 代表位置：`storage/experience.py:121, 278, 314, 323-348, 361`，`storage/evidence.py:69-86`，`workers/worker.py:497-507`。
- 影响：状态集合和状态机约束分散，拼写错误只能在运行期暴露。
- 建议：对有生命周期的值统一使用枚举；纯数据库判别值至少集中为常量。

### P1-9：`AuditLogger.close()` 的参数是误导性接口

- 位置：`observability/audit.py:179-180`
- 现象：接收 `timeout_ms` 后立即 `del timeout_ms`，实现没有等待或超时语义；`NullAuditLogger.close()` 又保留相同参数。
- 影响：调用者会误以为关闭过程受超时控制。
- 建议：删除参数，或真正实现有界等待；若是为协议兼容保留，应在 docstring 中明确“当前同步关闭，参数仅兼容”。

## P2（可以优化）

### P2-1：重复的小工具函数可适度收敛

- `_now()` 至少出现在 `api/server.py`、`application/ingest.py`、`application/recall.py`、`observability/audit.py`、`workers/deduplicate.py`、`workers/worker.py`。
- `_clamp()` 在 `recall/ranking.py:11-15` 与 `recall/relation_expansion.py:79-83` 完全重复。
- 建议：仅收敛具有一致语义的纯函数；避免为一行 `_now()` 引入重量级公共层。

### P2-2：命名中仍有兼容别名和职责错位

- `Reranker = DashScopeReranker` 同时出现在 `components.py:33` 与 `recall/reranker.py:86`。
- `InvalidStateTransitionError` 实际是 `InvalidTransitionError` 的别名。
- `ExperienceService` 实际是 repository 子类。
- 建议：确定唯一公开名称；兼容别名标注 deprecation 版本，避免长期双名。

### P2-3：行宽和格式化尚未完全统一

- 共有 9 行超过项目约定的 120 字符。
- 最明显的是 `ingest/llm_extractor.py:56`，单行 354 字符。
- 其余集中在 `storage/experience.py:168, 323, 347, 374, 376` 和 migration CLI。
- 建议：在开发依赖中加入 Black/Isort/Ruff 并配置 CI；当前 `pyproject.toml` 尚无这些工具配置。

### P2-4：SQL 排版风格不一致

- 有的 SQL 关键字大写、分行清晰；有的把多个赋值和 `CASE` 压在一个长字符串中，例如 `storage/experience.py:323-348`。
- 有的单引号字面量直接嵌入 SQL，有的使用参数。
- 建议：统一多行 SQL 的缩进与关键字风格；动态值继续参数化，固定 schema 判别值保持一致写法。

### P2-5：部分默认值仍直接存在于组件构造器

- `recall/reranker.py:26-28`：URL、模型、timeout。
- `adapters/hermes/provider.py:19-22`：daemon URL、timeout。
- `ingest/budget.py:29`：SQLite timeout。
- Settings 已覆盖其中大部分，但直接构造类时会绕过集中配置。
- 建议：生产组件构造器要求显式传入运行时参数；只在 Settings 层保留默认值。

### P2-6：若干返回类型仍过于宽泛

- `api/schemas.py:71` 的 `ClaimOutput.text: Any`。
- 多个 API 返回 `dict[str, Any]`，尽管已存在 Pydantic DTO。
- `protocols.py` 的搜索结果是裸 `list[dict]`。
- 建议：对外输出优先声明 DTO，对内记录优先 TypedDict；`Any` 只保留在真正不可知的 JSON value 叶子。

### P2-7：请求体限制中间件的名称和实际保证不完全一致

- 位置：`api/server.py:51-63`
- 现象：只检查可选的 `Content-Length`；没有该 header（例如流式/chunked 请求）时不会实际计量 body，非法整数也没有映射为明确的 4xx。
- 影响：`RequestSizeLimitMiddleware` 这个名称暗示了比实现更强的保证。
- 建议：改名为 `ContentLengthLimitMiddleware`，或实现对实际接收字节数的限制。

## 结构与一致性补充

- 未发现静态循环 import。
- `domain` 与 `core` 基本没有反向依赖基础设施，分层方向良好。
- migration snapshot 中的大段重复领域逻辑属于不可变历史快照，不建议为消除重复而回改；它应在重复统计中豁免。
- 包级 `__init__.py` 的显式 re-export（例如 `llm/__init__.py`、`application/__init__.py`）是公开 API，不应当按普通未使用 import 删除。
- 多数宽泛 `except Exception` 位于“rollback 后重新抛出”的事务边界，属于合理用法；本报告没有把这些机械列为问题。

## 建议修复顺序

1. 修复 P0-1、P0-2，并增加针对空 query/trigger 与控制流异常的回归测试。
2. 统一 reranker/Hermes 的 HTTP retry 与失败语义。
3. 清理确定死代码和无意义测试工厂。
4. 用 TypedDict/Protocol 收紧 recall、ingest、storage 三条主链路，先消除 `X | Any` 和裸 `list[dict]`。
5. 抽取日调度公共逻辑，再拆分 ingest、relation expansion、settings validation。
6. 补齐公开 docstring，并接入 Black/Isort/Ruff/Mypy 或 Pyright 的 CI 门禁。
