# hl_mem v0.4.3 架构复杂度审查任务

## 背景

hl_mem 是一个本地 Agent 记忆系统（Python，SQLite 后端），刚完成 Phase 13（13个架构问题修复）+ Phase 14（Hindsight 对标改进：LLM Provider 解耦、Pydantic schema、长输入分块、SearchTrace、一跳关系扩展）。

当前版本 0.4.3，77 个 .py 源文件，250 测试（232 passed）。

## 审查目标

**核心问题：代码架构是否引入了过于复杂的抽象/分层/模式，导致后续扩展困难、维护成本升高？**

请从以下 6 个维度审查，每个问题按 P0（严重/必须修）/ P1（重要/应该修）/ P2（建议/可选）分级：

### 1. 抽象层次是否过度
- 是否有不必要的间接层？（如一层薄 wrapper 只是转发调用）
- 是否有提前抽象（只有一个实现的接口/协议）？
- 是否有过度参数化（函数有大量可选参数但实际调用从不传）？
- LLMClient → Provider 的分层是否引入了不必要的复杂度？

### 2. 模块划分是否合理
- 模块边界是否清晰？有没有职责重叠/碎片化的模块？
- 新增的 llm/、ingest/、recall/ 子包是否让代码更难导航？
- 是否有"上帝模块"（单文件过大，承担太多职责）？
- 是否有"纳米模块"（单函数文件，不值得独立成模块）？

### 3. 依赖关系是否健康
- 有没有不必要的循环依赖风险？
- 导入链是否过长？（A→B→C→D 才能用一个功能）
- 是否有"黑洞模块"（被太多其他模块依赖，改一处影响全局）？
- __init__.py 是否有过度 re-export 导致的隐式依赖？

### 4. 数据模型一致性
- Pydantic schema 约束是否与实际使用匹配？有没有过度严格的验证导致灵活性丧失？
- 数据流中是否有多次不必要的格式转换（dict → model → dict → model）？
- 配置项（环境变量/config）是否散落各处难以管理？

### 5. 可维护性
- 新人接手需要理解多少层才能做一个简单的修改？
- 测试是否过度耦合实现细节（如 monkeypatch 模块内部函数）？
- 是否有足够的文档/docstring 解释"为什么这样设计"而非"做了什么"？
- magic number/常量是否集中管理？

### 6. 扩展空间
- 当前架构如果要加新 Provider（如 Ollama 本地模型），需要改几个文件？
- 如果要加新的记忆类型（如 episodic memory），需要侵入式改动吗？
- 如果要做多租户隔离，当前架构阻碍在哪里？
- 一跳关系扩展如果要升级到多跳，设计是否支持平滑过渡？

## 审查要求

1. **读完全部 src/hl_mem/ 下的源码**（77 个 .py 文件），不要只看目录结构猜测
2. **具体到文件:行号**，每个问题给出精确位置
3. **按实际场景评估**：这是一个本地单 Agent 记忆系统（非企业级 SaaS），不要套用微服务/多租户标准
